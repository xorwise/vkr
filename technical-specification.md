# Техническое задание на разработку web-приложения для работы с онтологией маршрутов городского транспорта

## 1. Общие сведения

**Наименование разработки:** `Web-приложение для работы с онтологией маршрутов городского транспорта`.

**Основание для разработки:**
разработка выполняется в рамках выпускной квалификационной работы.

**Область применения:**
информационные системы визуализации и анализа данных о маршрутах городского транспорта, остановках и связанных объектах городской инфраструктуры.

## 2. Назначение системы

Разрабатываемое приложение предназначено для загрузки, обработки, визуализации и анализа данных онтологии городского транспорта. Система должна предоставлять пользователю доступ к сведениям о маршрутах, остановках, видах транспорта и объектах городской инфраструктуры через web-интерфейс с картографической основой.

## 3. Цель разработки

Целью разработки является создание web-приложения, обеспечивающего наглядную работу с онтологией маршрутов городского транспорта, включая поиск, просмотр, фильтрацию, анализ связей между сущностями и визуализацию маршрутов на электронной карте.

## 4. Задачи разработки

В рамках разработки необходимо решить следующие задачи:

1. Реализовать хранение и обработку данных предметной области на основе онтологии.
2. Реализовать импорт и разбор онтологических данных из файла формата OWL/XML.
3. Реализовать серверную часть приложения для предоставления доступа к данным через HTTP API.
4. Реализовать клиентскую часть приложения с интерактивным web-интерфейсом.
5. Реализовать отображение маршрутов, остановок и объектов инфраструктуры на карте.
6. Реализовать поиск остановок и фильтрацию маршрутов по видам транспорта.
7. Реализовать выбор маршрута с отображением его состава и остановок.
8. Реализовать построение маршрута на карте с использованием картографического API Yandex Maps.
9. Реализовать режим фокусировки, при котором при выборе маршрута отображается только выбранный маршрут и связанные с ним остановки.
10. Обеспечить корректную обработку ошибок загрузки данных и взаимодействия с API.

## 5. Объект и предмет автоматизации

**Объект автоматизации:**
процесс представления и анализа данных о маршрутах городского транспорта.

**Предмет автоматизации:**
данные об остановках, маршрутах, транспортных средствах и связанных объектах городской инфраструктуры, представленные в виде онтологии.

## 6. Функциональные требования

Система должна обеспечивать выполнение следующих функций:

1. Загрузка онтологии маршрутов городского транспорта из файла.
2. Выделение из онтологии сущностей: остановки, маршруты, транспорт, объекты городской инфраструктуры.
3. Выделение и обработка связей между сущностями: маршрут - транспорт, маршрут - остановки, здание - ближайшая остановка.
4. Предоставление API для получения списка остановок.
5. Предоставление API для получения списка маршрутов.
6. Предоставление API для получения данных о конкретной остановке.
7. Предоставление API для получения данных о конкретном маршруте.
8. Предоставление API для поиска остановок по названию.
9. Отображение карты города в web-интерфейсе.
10. Отображение остановок на карте.
11. Отображение объектов городской инфраструктуры на карте.
12. Отображение маршрутов на карте.
13. Выбор маршрута из списка или по клику на карте.
14. Отображение сведений о выбранном маршруте: номер, тип транспорта, список остановок, описание маршрута.
15. Построение выбранного маршрута средствами Yandex Maps API.
16. Скрытие лишних маршрутов и объектов при выборе конкретного маршрута для уменьшения визуального шума.
17. Поиск остановки по текстовому запросу.
18. Просмотр информации об остановке: название, координаты, маршруты через остановку, близлежащие объекты.
19. Фильтрация маршрутов по виду транспорта.
20. Закрытие текущего выбранного маршрута и возврат к общей карте.

## 7. Дополнительные функциональные требования

В ТЗ включаются также дополнительные возможности, повышающие практическую ценность системы:

1. Валидация входной онтологии перед загрузкой в систему.
2. Вывод сообщений о неполных или некорректных данных онтологии.
3. Автоматическое геокодирование остановок и объектов инфраструктуры при отсутствии координат.
4. Поиск объектов городской инфраструктуры рядом с выбранной остановкой.
5. Адаптация интерфейса для экранов мобильных устройств.
6. Поддержка легенды карты и цветового различия видов транспорта.
7. Возможность масштабирования карты и центрирования на выбранном маршруте.
8. Возможность повторного использования API другими клиентскими приложениями.
9. Логирование основных действий системы и ошибок обработки данных.
10. Подготовка системы к расширению предметной области: добавление новых видов транспорта, новых классов объектов и новых онтологических связей.
11. Возможность экспорта сведений о маршруте в структурированном виде.
12. Возможность вывода сводной статистики по маршрутам, остановкам и объектам.
13. Подготовка архитектуры к добавлению административного режима загрузки новой версии онтологии.

## 8. Требования к входным данным

Входными данными системы являются:

1. Файл онтологии в формате OWL/XML.
2. Данные картографического сервиса Yandex Maps.
3. Поисковые запросы пользователя.
4. Параметры фильтрации маршрутов и остановок.

Входной файл должен содержать:

1. Описание классов предметной области.
2. Описание индивидов.
3. Объектные свойства между сущностями.
4. Свойства данных, включая названия и иные атрибуты.

## 9. Требования к выходным данным

Система должна формировать и предоставлять пользователю:

1. Список маршрутов.
2. Список остановок.
3. Сведения о выбранном маршруте.
4. Сведения о выбранной остановке.
5. Сведения о близлежащих объектах инфраструктуры.
6. Картографическое отображение маршрутов и остановок.
7. Результаты поиска и фильтрации.
8. Сообщения об ошибках и некорректных входных данных.

## 10. Требования к программной архитектуре

Система должна иметь клиент-серверную архитектуру.

Серверная часть должна:

1. Быть реализована на языке Go.
2. Обеспечивать разбор онтологии.
3. Обеспечивать хранение данных в оперативной памяти.
4. Предоставлять REST API для клиентской части.

Клиентская часть должна:

1. Быть реализована как одностраничное web-приложение.
2. Обеспечивать взаимодействие с серверным API.
3. Обеспечивать визуализацию данных на карте.
4. Обеспечивать интерактивную работу пользователя с маршрутами и остановками.

Для картографической визуализации должно использоваться API Yandex Maps.

## 11. Нефункциональные требования

К системе предъявляются следующие нефункциональные требования:

1. Интерфейс должен быть понятным и наглядным.
2. Система должна обеспечивать приемлемую скорость отклика при работе со списками маршрутов и остановок.
3. Время загрузки основных данных после запуска приложения не должно быть избыточным для пользовательского сценария.
4. Система должна корректно работать в современных браузерах.
5. Система должна обеспечивать отказоустойчивую обработку ошибок API и ошибок загрузки данных.
6. Код должен иметь модульную структуру и быть пригодным для дальнейшего сопровождения.
7. Интерфейс должен быть адаптирован под различные разрешения экрана.
8. Система должна обеспечивать читаемое визуальное выделение выбранного маршрута.

## 12. Требования к пользовательскому интерфейсу

Интерфейс приложения должен содержать:

1. Боковую панель со списком маршрутов.
2. Строку поиска остановок.
3. Элементы фильтрации по видам транспорта.
4. Основную область карты.
5. Панель сведений о выбранном маршруте.
6. Всплывающее окно или карточку сведений о выбранной остановке.

Интерфейс должен обеспечивать:

1. Выбор маршрута из списка.
2. Выбор маршрута на карте.
3. Просмотр остановок выбранного маршрута.
4. Просмотр объектов рядом с остановкой.
5. Закрытие текущего выбора и возврат к исходному режиму просмотра.

## 13. Требования к надежности

Система должна:

1. Корректно обрабатывать отсутствие части данных в онтологии.
2. Сохранять работоспособность при отсутствии координат у отдельных объектов.
3. Выводить понятные сообщения при невозможности загрузить данные или построить маршрут.
4. Не приводить к аварийному завершению при ошибках внешних сервисов.

## 14. Требования к тестированию

В процессе разработки необходимо провести:

1. Тестирование загрузки и разбора онтологии.
2. Тестирование API-методов.
3. Тестирование поиска остановок.
4. Тестирование отображения маршрутов на карте.
5. Тестирование выбора маршрута и отображения только выбранного маршрута.
6. Тестирование отображения сведений об остановках.
7. Тестирование адаптивности интерфейса.
8. Тестирование обработки ошибок.

## 15. Критерии приемки

Система считается соответствующей ТЗ, если:

1. Успешно загружает и обрабатывает онтологию.
2. Предоставляет пользователю сведения о маршрутах, остановках и объектах.
3. Корректно отображает маршруты на карте.
4. Позволяет искать остановки и фильтровать маршруты.
5. Отображает только выбранный маршрут в режиме фокусировки.
6. Позволяет получить сведения о выбранной остановке.
7. Корректно взаимодействует с Yandex Maps API.
8. Работает без критических ошибок в целевых браузерах.

## 16. Перспективы развития

В дальнейшем система может быть расширена за счет:

1. Поддержки нескольких городов.
2. Подключения базы данных.
3. Редактирования онтологии через web-интерфейс.
4. Добавления модуля аналитики транспортной сети.
5. Построения пользовательских сценариев поездок.
6. Интеграции с другими геоинформационными сервисами.
